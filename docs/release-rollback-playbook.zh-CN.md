# ForgePilot 发布回滚手册

## 目标

在发布后出现故障时，保证可以在 30 分钟内将服务回退到上一稳定版本，并保留审计证据。

## 回滚触发条件

1. 关键路径（任务执行、文件修改、验证报告）连续失败率高于 20%。
2. 生产错误在 10 分钟内持续增长，且无法通过配置热修快速恢复。
3. 出现数据一致性风险或安全风险。

## 回滚步骤

1. 冻结变更

- 暂停当前发布批次的后续部署与数据库迁移。
- 在团队频道同步回滚窗口和负责人。

2. 回退应用版本

- 将后端、前端和运行时镜像 tag 切回上一个稳定 tag。
- 若使用 Helm，执行同版本 `helm rollback`。

3. 回退配置

- 还原本次发布新增或变更的环境变量。
- 若涉及命名迁移，按 `scripts/forgepilot-rename-rollback.patch` 回退对应改名批次。

4. 回退数据变更（如有）

- 按迁移说明执行数据库 downgrade 或恢复备份快照。
- 校验租户、任务、审计日志等关键表的数据一致性。

5. 验证与复盘

- 执行最小验收链路：创建任务 -> 执行命令 -> 修改文件 -> 验证 -> 交付报告。
- 导出回滚窗口的审计日志，归档到对象存储。
- 在 `CHANGELOG.md` 标注“已回滚”并补充原因。

## 角色分工

- 发布负责人：发起回滚、协调沟通、确认恢复。
- 运维负责人：执行镜像与基础设施回退。
- 后端负责人：处理数据与接口回退。
- 前端负责人：验证任务台与审计回放可用性。
